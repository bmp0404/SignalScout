# Services

This module holds application-level orchestration classes that sit above repositories/scoring/discovery/enrichment/digest and are what the API layer (backend/api/routes.py) actually calls, wired together in backend/container.py.

## backend/services/__init__.py
Empty package marker file with no exported symbols.

## backend/services/candidate_review.py
One-click workflow rules for sorting discovery candidates into unreviewed / approved / rejected.

- `CandidateReviewService` ŌĆö orchestrates recording review decisions on discovery candidates, backed by `CandidateReviewRepository`, `PersonRepository`, and `SignalRepository`.
  - `CandidateReviewService.review(person_id, state, why_now, notes, source_bucket, contactable, primary_evidence_url, reviewer) -> CandidateReview` ŌĆö loads the person (must exist and be in the "discovery" cohort), then upserts the review record; when `state == "approved"` and `contactable` was not set, defaults `contactable=True` so one-click approves flow into the digest.
  - `CandidateReviewService.list_rows(state=None) -> list[dict]` ŌĆö fetches reviews (optionally filtered by state), joins each to its person, and returns dicts merging review fields with the person's name and display contacts.
  - `CandidateReviewService.approved_mix() -> dict[str, int]` ŌĆö counts approved, contactable reviews by `source_bucket`, returned sorted by bucket name.

## backend/services/candidate_service.py
Scores the live cohort and builds the payloads the UI needs ŌĆö score receipts, connection context, warm-intro paths, and "why now" lines.

- `CandidateService` ŌĆö orchestrates scoring and candidate-facing read views, composing `PersonRepository`, `SignalRepository`, `GraphEdgeRepository`, `ScoringEngine`, and (optionally) `CandidateReviewRepository`.
  - `CandidateService.__init__(persons, signals, edges, engine, flag_threshold, reviews=None)` ŌĆö stores the repositories, scoring engine, the score threshold used to flag standout candidates, and an optional reviews repository for review-state enrichment.
  - `CandidateService.rescore_all() -> dict[str, float]` ŌĆö for every founder/discovery/demo person, builds their signals plus a synthetic connection signal to founder "seeds" (signals and edges are batch-loaded once via `signals.for_people`/`edges.for_people`), computes a base score via `ScoringEngine.compute`, applies a knownness down-weight, calibrates all scores against a founder reference pack (`founder_reference`) via `ScoringEngine.normalize_calibrated`, persists them in one transaction with `persons.update_scores`, and returns the normalized score map.
  - `CandidateService._knownness_factor(person) -> float` (static) ŌĆö down-weights already-famous GitHub accounts (by follower count bands: <=500 ŌåÆ 1.0, <=1000 ŌåÆ 0.7, <=2000 ŌåÆ 0.5, else 0.3) so ranking favors pre-breakout people; founders/seeded profiles without a follower count are unaffected.
  - `CandidateService.list_candidates(cohort="discovery") -> list[dict]` ŌĆö lists scored people in a cohort (or all non-control people if `cohort` is falsy), sorted by score descending, each rendered via `_summary`; signals, edges, and reviews for the whole page are batch-loaded once (`signals.for_people`/`edges.for_people`/`reviews.for_people`) rather than per person.
  - `CandidateService.profile(person_id) -> dict | None` ŌĆö builds a full candidate profile: signals plus a synthetic founder-connection signal, a score breakdown, a chronological signal timeline, founder connections, and contact info, layered on top of `_summary`.
  - `CandidateService._summary(person, founders_by_id, sigs, person_edges, review) -> dict` ŌĆö assembles the compact candidate card from already-loaded signals/edges/review (so both `list_candidates` and `profile` supply them from batch or single loads): identity/location/school fields, top 3 signals, source counts/diversity, GitHub follower count, founder-connection count/context/warm-intro, why-now text (preferring a human-reviewed one), review/approval state, contact links, evidence coverage, and discovery/enrichment status (`discovery_origin`, `discovery_source`, `evidence_status`/`evidence_tier`, `review_required`, `enrichment_status`, `enrichment_provider`).
  - `CandidateService._connection_list(person, edges, founders_by_id) -> list[dict]` ŌĆö turns edges linking the person to known founders into human-readable connection descriptions (via `EDGE_VERBS`), sorted by observed date.
  - `CandidateService._seed_edges(person, edges, founders_by_id) -> list[GraphEdge]` ŌĆö filters a person's edges down to those connecting them to a founder ("seed").
  - `CandidateService._connection_context(seed_edges, founders_by_id) -> str` (static) ŌĆö summarizes founder connections into one sentence, grouping "follows" edges together and listing up to two other relationship types.
  - `CandidateService._warm_intro(person, seed_edges, founders_by_id) -> str` (static) ŌĆö picks the highest-quality founder edge (by `EDGE_QUALITY`) and phrases a "reach out via ..." suggestion.
  - `CandidateService._why_now(signals) -> str` (static) ŌĆö generates a one-line rationale: prefers a signal showing a >=1.8x star-count jump, else describes the most recent signal, phrased differently depending on whether it's within the last year.
  - `CandidateService._source_counts(signals) -> dict[str, int]` (static) ŌĆö counts signals per source, sorted descending by count.
  - `CandidateService._coverage(signals) -> str` (static) ŌĆö classifies evidence coverage as HIGH (3+ distinct sources), MED (2), or LOW (0-1).
  - `CandidateService._breakdown_dict(breakdown) -> dict` (static) ŌĆö converts a `ScoreBreakdown` dataclass into a plain dict (raw, items, diversity multiplier, recency bonus, age factor, adjusted score).

## backend/services/discovery_job.py
`DiscoveryJobManager` runs the live discovery pipeline in a background thread and exposes in-memory stage progress for polling. This is the "OG" discovery pipeline (`POST /api/discovery/run`) ŌĆö it always runs `ProviderExpander.expand()` in batch mode over every filter set in `data/provider_discovery_filters.json`; it does not know about `DiscoveryRecipe`s, which are run separately and synchronously via `discovery_recipe_service.py`.

- `DiscoveryJobManager` ŌĆö orchestrates a single global, threaded discovery run through four UI-facing stages (Scrape -> Resolve -> Enrich -> Score), building its own `Container` (own SQLite connection) per run so writes never collide with the API's read connection.
  - `DiscoveryJobManager.__init__(settings, container_factory)` ŌĆö stores settings, a factory that builds a fresh `Container`, a `threading.Lock`, and initializes idle in-memory state.
  - `DiscoveryJobManager._idle_state() -> dict` (static) ŌĆö returns the initial/reset job-state dict (job id, state, per-stage status/counts, per-source discovery counts, timestamps, error).
  - `DiscoveryJobManager.status() -> dict` ŌĆö returns a deep copy of the current in-memory job state (cheap, DB-free, for polling).
  - `DiscoveryJobManager.start() -> str` ŌĆö guards against a concurrently running job, requires at least one configured discovery lane (GitHub token or a provider chain), then generates a job id, resets state to "running", and launches `_run` on a daemon thread.
  - `DiscoveryJobManager._resolve_seeds(container) -> list[str]` ŌĆö prefers curated `demo_seeds` from the seed-accounts file that already exist in the DB, else falls back to founder `github_seeds`; optionally appends fellowship-alumni seeds via `FellowshipSeedLoader`; truncates to `discovery_seed_limit`.
  - `DiscoveryJobManager._set_stage(name, status=None, count=None)` ŌĆö thread-safely updates one pipeline stage's status/count in the shared state.
  - `DiscoveryJobManager._set_source_count(source, count)` ŌĆö thread-safely updates a per-source discovery count.
  - `DiscoveryJobManager._run_provider_lane(container) -> list` ŌĆö runs budgeted provider-search discovery (PDL -> Coresignal) via `container.provider_expander.expand` (batch mode), records per-source counts, logs a summary, and returns newly-created discovery people.
  - `DiscoveryJobManager._run_openalex_lab_lane(container) -> list` ŌĆö opt-in (`settings.discovery_include_openalex`) curated-lab lead-gen via `container.openalex_lab_expander.expand`, logs a summary, and returns newly-created discovery people; a no-op returning `[]` when the flag is off.
  - `DiscoveryJobManager._run_github_lane(container, token, on_progress) -> tuple[GithubScraper, list]` ŌĆö resolves seeds, builds a `GithubScraper`/`GraphExpander`, and expands from seeds (capped per-seed/per-repo/per-org limits for a short on-camera run), returning the scraper and discovered people.
  - `DiscoveryJobManager._save_collected(container, signals, edges)` (static) ŌĆö resolves and persists a batch of signals and edges via `container.resolver` and the signals/edges repositories.
  - `DiscoveryJobManager._run_collaboration_lane(container, github, fresh_people) -> list[Person]` ŌĆö collects capped Semantic Scholar (up to 8, both co-authorship via `collect()` and citation edges via `collect_citations()`, sharing one resolved author lookup), OpenAlex (up to 8, co-authorship via `collect()`), and Devpost (up to 8) collaboration evidence for fresh plus existing discovery people, then runs `CollaborationExpander.expand` (using the GitHub scraper, Devpost/Scholar/OpenAlex scrapers, and `provider_enricher`) to promote dead-end/collaboration/citation hits into discovery people, recording per-source counts.
  - `DiscoveryJobManager._run(job_id)` ŌĆö the pipeline entrypoint executed on the background thread: builds a Container, runs the provider lane first (primary source, needs no GitHub account), then the opt-in OpenAlex lab lane (also no GitHub account needed), then the GitHub lane if a token is configured, then the collaboration lane on the combined fresh people; marks scrape/resolve done; enriches each GitHub/collaboration person's contact info and location (scraping fresh signals for GitHub-sourced people) and saves them; runs the provider-enrichment queue (`container.provider_enricher.prioritize`/`.run`) over pending discovery people with a GitHub username; marks enrich done; calls `container.candidate_service.rescore_all()` and marks score done; on any exception, logs it, marks the job "error" with the exception message, and flips any still-active stage to "error"; always closes the container's DB connection in a `finally` block.

## backend/services/discovery_recipe_service.py
`DiscoveryRecipeService` is the "recipe" discovery pipeline's API-facing orchestrator: list/run/dry-run/approve `DiscoveryRecipe`s and summarize spend. Orchestration only ŌĆö every search, dedupe, and credit spend happens inside `ProviderExpander`/`ProviderBudget`/`provider_search_checkpoints` (see `discovery.md`, `enrichment.md`, `db.md`); this service adds no parallel dedupe ledger or budget table, and shares those with `DiscoveryJobManager`'s batch pipeline.

- `DiscoveryRecipeService` ŌĆö composes `DiscoveryRecipeRepository`, `ProviderIdentityRepository`, `ProviderExpander`, `ProviderBudget`, `EnrichmentUsageRepository`, and `PersonRepository` (plus optional `candidate_service` for post-run rescoring).
  - `DiscoveryRecipeService.__init__(recipes, identities, expander, budget, usage, persons, candidate_service=None, per_recipe_cap=None)` ŌĆö stores the repositories/services used for listing, running, and cost reporting; `per_recipe_cap` (from `Settings.provider_per_recipe_cap`) bounds how many records any single recipe may pull per scheduled run so one recipe can't drain a provider's shared daily budget.
  - `DiscoveryRecipeService.list_recipes() -> list[dict]` ŌĆö returns `_recipe_row(recipe)` for every stored recipe (includes a `due` flag).
  - `DiscoveryRecipeService.approve(recipe_id) -> dict` ŌĆö sets a recipe's `approval_state` to `"approved"` (raises `ValueError` if the recipe id is unknown) and returns its updated row.
  - `DiscoveryRecipeService.run(recipe_id, override_limit=None) -> dict` ŌĆö real (non-dry) recipe run; delegates to `_run`.
  - `DiscoveryRecipeService.dry_run(recipe_id, override_limit=None) -> dict` ŌĆö dry-run recipe preview; delegates to `_run`.
  - `DiscoveryRecipeService.is_due(recipe, now=None) -> bool` ŌĆö true when the recipe is `active`, `approved`, has a weekly/biweekly frequency (not `manual`), and `last_run` is missing or older than the interval.
  - `DiscoveryRecipeService.run_due(now=None) -> dict` ŌĆö runs every due recipe (each capped at `per_recipe_cap` records via `override_limit`, continues on per-recipe errors), optionally rescores via `candidate_service` when anyone was created, and returns aggregate stats (`due_count`, `ran_count`, `error_count`, `created_total`, `results`).
  - `DiscoveryRecipeService.cost_summary() -> dict` ŌĆö builds the operator cost dashboard payload: per-provider `search_credits_used`/`search_credits_remaining` (PDL counted monthly, Coresignal daily, matching `ProviderBudget`'s own period semantics), per-recipe totals (credit units, the Search/Collect split, created/duplicate/merged counts) by calling `expander.recipe_checkpoint(recipe)` for every stored recipe, a running `duplicates_skipped` total (`duplicate_count + merged_count` summed across checkpoints), an `enrichment_credits_saved` estimate (every search-discovered candidate arrived fully enriched, so each one is an ENRICH-lane credit that was never spent ŌĆö approximated as the sum of `verified_count + review_count` across checkpoints), and `candidates_by_discovery_source` (via `_candidates_by_discovery_source`).
  - `DiscoveryRecipeService._run(recipe_id, dry_run, override_limit) -> dict` ŌĆö looks up the recipe (raises `ValueError` if unknown), calls `expander.run_recipe(recipe, dry_run=dry_run, override_limit=override_limit)` (which itself raises `PermissionError` for an unapproved real run ŌĆö left to propagate to the API layer), and advances `recipes.set_last_run` **only on a real run that reached the provider without being a pure error** (`reached_provider = produced or (attempted and not errors)`, where `produced` = any created/returned/credit activity) so a missing key, exhausted budget, or a 422/5xx error run stays due instead of silently skipping the recipe's next window; returns a flat summary dict (recipe id/provider, `provider_configured`, `reached_provider`, `skip_reason` (`provider_not_configured`/`budget_exhausted`/`up_to_date`/`None`), dry_run flag, attempted/returned/created/verified/review/merged/duplicates/`errors`/rejected/rejection_reasons/credit_units/planned_pages).
  - `DiscoveryRecipeService._recipe_row(recipe) -> dict` ŌĆö joins a `DiscoveryRecipe`'s stored fields with its current `recipe_checkpoint` (last result count, last created count, last duplicate count, last credit units, last outcome ŌĆö all `0`/`"never_run"` if no checkpoint exists yet) plus `provider_configured` (whether the recipe's provider key is set) and `due` for the `GET /api/discovery/recipes` list view.
  - `DiscoveryRecipeService._get(recipe_id) -> DiscoveryRecipe` ŌĆö fetches a recipe or raises `ValueError` if the id is unknown.
  - `DiscoveryRecipeService._candidates_by_discovery_source() -> dict[str, int]` ŌĆö counts discovery-cohort people by `discovery_source` (falling back to `"unspecified"` when unset), for the Pipeline page's source-mix chart.

## backend/services/discovery_scheduler.py
In-process background ticker that periodically calls `DiscoveryRecipeService.run_due()` so Discover stays populated without manual Pipeline clicks.

- `DiscoveryScheduler` ŌĆö daemon-thread scheduler keyed off `Settings.discovery_background` / `discovery_background_interval_hours`.
  - `DiscoveryScheduler.__init__(settings, container_factory)` ŌĆö stores settings and a factory that builds a fresh `Container` per tick (thread-local DB connections).
  - `DiscoveryScheduler.start() -> None` ŌĆö no-op when background is disabled; otherwise starts the daemon loop.
  - `DiscoveryScheduler.stop() -> None` ŌĆö signals the loop to exit and joins briefly.
  - `DiscoveryScheduler._loop() -> None` ŌĆö waits 15s, then repeatedly ticks and sleeps the configured interval hours.
  - `DiscoveryScheduler._tick() -> None` ŌĆö builds a container, calls `run_due()`, logs summary, closes the DB; swallows exceptions so one failure does not kill the loop.

## backend/services/subscriber_digest.py
Builds and delivers personalized, never-repeat subscriber digests (email HTML/text), drives the scheduled digest run, and builds the operator-facing "upcoming digest" preview. Module constants `FREQUENCY_INTERVALS` (daily=1d, every_3_days=3d, weekly=7d), `DEFAULT_FREQUENCY` (`every_3_days`), and `DEFAULT_INTERVAL` define the cadence.

- `SubscriberDigestService` ŌĆö orchestrates digest content selection, rendering, and delivery, composing `SubscriberRepository`, `DigestSendRepository`, `CandidateService`, `EmailSender`, and `EmailActionSigner`.
  - `SubscriberDigestService.__init__(subscribers, sends, candidates, sender, public_base_url, action_signer, size=10)` ŌĆö stores the repositories/services/sender, the public base URL (used for feedback/unsubscribe links), the action signer, and the default digest size (10 picks).
  - `SubscriberDigestService.build(subscriber, all_candidates=None) -> tuple[EmailMessage, list[str]]` ŌĆö selects the subscriber's picks via `_select_picks` (excluding people already sent to them via `sends.sent_person_ids`, with subscriber-preference sorting) and renders both an `EmailMessage` (marking provisional backfill picks) and the list of picked person ids. `all_candidates` lets a multi-subscriber send compute the (subscriber-independent) discovery pool once and reuse it; when omitted it falls back to a per-call `list_candidates("discovery")`.
  - `SubscriberDigestService._select_picks(candidates, exclude_ids, subscriber=None) -> tuple[list[dict], set[str]]` ŌĆö primary pool is approved+contactable candidates not in `exclude_ids`, newest approvals first (then subscriber preference/score); if short of `size`, backfills with top-scored verified-tier candidates that have >=2 contact links, returning those backfilled ids as "provisional".
  - `SubscriberDigestService._has_contacts(candidate) -> bool` (static) ŌĆö true when a candidate exposes at least two of github/linkedin/x/email/site links (backfill contactability gate).
  - `SubscriberDigestService.upcoming(offset=0) -> dict` ŌĆö operator/Cory-facing preview of the digest lineup, paginated so each Refresh advances to a fresh batch: rescores, orders the approved+contactable pool with not-yet-featured people first (`sends.all_sent_person_ids`), then returns the `size`-person window starting at `offset` and wrapping around the pool so repeated refreshes cycle through everyone. Unlike a per-subscriber email it does not hard-exclude featured people, keeping the tab full; verified-tier candidates with 2+ contacts backfill remaining slots (flagged `provisional`) when the approved pool is smaller than `size`. Returns `entries`, `auto_send` status, `featured_count`, `pool_size`, `offset`, and `next_offset` (what the client passes next). Backs `GET /api/digest/upcoming`.
  - `SubscriberDigestService._auto_send_status() -> dict` ŌĆö active-subscriber count, last send timestamp (`sends.last_sent_at`), `default_cadence`, and a per-cadence subscriber count map.
  - `SubscriberDigestService._entry(candidate, provisional) -> dict` (static) ŌĆö shapes a candidate into the frontend digest-entry dict (person_id, name, score, school/location lines, thesis, top signals, orbit/intro/why-now, contact links, `provisional`).
  - `SubscriberDigestService.preview(subscriber) -> dict` ŌĆö calls `build`, re-hydrates the picked candidates' full dicts, computes a source-bucket mix, and returns subject/html/text plus candidate details for preview purposes (no send, no record).
  - `SubscriberDigestService.deliver(subscriber, dry_run=False, all_candidates=None) -> dict` ŌĆö calls `build` (forwarding the shared `all_candidates` pool); if there are no picks, returns an "empty" status; if `dry_run`, returns a "preview" payload without sending; otherwise sends via `self.sender.send`, records the sent person ids in `sends.record_many` on success, and returns a status of sent/preview/failed based on the sender's receipt. `run_due`/`send_to_active` compute the discovery pool once and pass it to every `deliver`.
  - `SubscriberDigestService.send_to_active(dry_run=False, now=None) -> dict` ŌĆö operator "send now": rescores the pool, then delivers to every active subscriber immediately (ignoring cadence) via the same per-subscriber `deliver` path (real Resend send with preview-only fallback, deduped so nobody is emailed twice); returns aggregate stats (`subscriber_count`, `sent_count`, `empty_count`, per-subscriber `results`). Backs `POST /api/digests/send` (admin-gated).
  - `SubscriberDigestService.run_due(dry_run=False, recipient=None, now=None) -> dict` ŌĆö rescores the pool, fetches active subscribers, and (when no specific `recipient` is given) delivers only to those whose cadence window has elapsed via `_is_due`; interval-based and idempotent, so re-ticking or a redundant external cron never double-sends. Returns aggregate run stats. Drives the in-process `DigestScheduler` and `POST /api/digest/cron`.
  - `SubscriberDigestService._is_due(subscriber, run_at) -> bool` ŌĆö true when the subscriber's most recent send is older than their cadence interval (`FREQUENCY_INTERVALS`, defaulting to `DEFAULT_INTERVAL`), via `sends.sent_since`.
  - `SubscriberDigestService._preference_rank(candidate, subscriber) -> tuple[int, float]` (static) ŌĆö scores how well a candidate matches the subscriber's `signal_interests` preference terms against the candidate's area/why-now/top-signal text, returning (match count, score) as a sort key.
  - `SubscriberDigestService._feedback_url(subscriber, person_id, vote) -> str` ŌĆö builds a signed feedback URL (via `action_signer.issue`) for a subscriber to vote up/down on a specific candidate in the digest.
  - `SubscriberDigestService._unsubscribe_url(subscriber) -> str` ŌĆö builds a signed unsubscribe URL for the subscriber.
  - `SubscriberDigestService._render_html(subscriber, picks, today, provisional_ids=None) -> str` ŌĆö renders the full HTML email body: per-candidate sections (score, name, context, why-now description, triggering signals, contact links, primary-evidence link, a provisional note for backfill picks, feedback up/down links) plus a header and unsubscribe footer.
  - `SubscriberDigestService._render_text(subscriber, picks, today, provisional_ids=None) -> str` ŌĆö renders the plain-text equivalent of the digest email with the same per-candidate content (including a provisional marker) and an unsubscribe line.

## backend/services/digest_scheduler.py
In-process background ticker that periodically calls `SubscriberDigestService.run_due()` so subscriber digests send automatically without an external cron.

- `DigestScheduler` ŌĆö daemon-thread scheduler keyed off `Settings.digest_background` / `digest_background_interval_hours`, mirroring `DiscoveryScheduler`.
  - `DigestScheduler.__init__(settings, container_factory)` ŌĆö stores settings and a factory that builds a fresh `Container` per tick (thread-local DB connections).
  - `DigestScheduler.start() -> None` ŌĆö no-op when `digest_background` is disabled; otherwise starts the daemon loop.
  - `DigestScheduler.stop() -> None` ŌĆö signals the loop to exit and joins briefly.
  - `DigestScheduler._loop() -> None` ŌĆö waits 20s (so the HTTP server binds first), then repeatedly ticks and sleeps the configured interval hours.
  - `DigestScheduler._tick() -> None` ŌĆö builds a container, calls `run_due()` (idempotent per cadence window), logs the due/sent counts, closes the DB; swallows exceptions so one failure does not kill the loop.
